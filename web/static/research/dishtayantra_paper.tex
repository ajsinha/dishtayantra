\begin{abstract}
Real-time data processing systems face fundamental challenges in achieving low-latency computation while supporting heterogeneous computational workloads across multiple programming languages. We present \textbf{DishtaYantra}, a novel high-performance Directed Acyclic Graph (DAG) compute framework that addresses these challenges through three key innovations: (1) an LMDB-based zero-copy data exchange mechanism achieving 100-1000$\times$ performance improvements over traditional serialization for inter-language communication, (2) a unified multi-language calculator framework supporting Python, Java (Py4J), C++ (pybind11), Rust (PyO3), and REST endpoints with sub-microsecond invocation overhead, and (3) a multiprocessing worker pool architecture with DAG affinity scheduling that bypasses Python's Global Interpreter Lock (GIL) for true CPU parallelism. Our experimental evaluation on consumer hardware (AMD Ryzen, 64GB RAM, Ubuntu 24.04) demonstrates that DishtaYantra achieves median latencies of 5$\mu$s for large payload transfers (compared to 500$\mu$s with JSON serialization), processes over 100,000 messages per second per worker, and maintains 90\% parallel efficiency up to 12 concurrent workers.

\textbf{Keywords:} Dataflow processing, DAG execution, Zero-copy communication, Multi-language integration, LMDB, Real-time systems, High-performance computing
\end{abstract}

\section{Introduction}

The proliferation of real-time data processing applications in financial services, IoT analytics, and machine learning pipelines has created unprecedented demand for high-performance dataflow frameworks \cite{akidau2013millwheel, carbone2015flink}. Modern enterprises require systems capable of processing millions of events per second with sub-millisecond latencies while supporting computational logic implemented in diverse programming languages optimized for specific tasks \cite{zaharia2016spark}.

Existing dataflow frameworks face three fundamental limitations:

\begin{enumerate}
    \item \textbf{Serialization overhead} in inter-process and inter-language communication introduces latencies of 1-50ms for large payloads, making them unsuitable for microsecond-sensitive applications \cite{murray2013naiad}.
    
    \item \textbf{Language heterogeneity} forces developers to choose between optimal algorithms (e.g., numerical computing in C++/Rust) and framework integration (typically Python/Java), resulting in suboptimal performance or complex polyglot architectures \cite{kleppmann2017designing}.
    
    \item \textbf{GIL limitations} in Python-based frameworks prevent true CPU parallelism, constraining throughput regardless of available hardware resources \cite{beazley2010gil}.
\end{enumerate}

We present \textbf{DishtaYantra}\footnote{Sanskrit for ``Computing Instrument'' -- reflecting the framework's role as a precision tool for data processing.}, a novel dataflow framework that addresses these limitations through three key innovations:

\begin{enumerate}
    \item \textbf{Zero-Copy Data Exchange:} An LMDB-based transport mechanism that achieves 100-1000$\times$ performance improvements by eliminating serialization overhead through memory-mapped file I/O shared across process boundaries.
    
    \item \textbf{Unified Multi-Language Support:} A calculator factory pattern supporting Python, Java (via Py4J), C++ (via pybind11), Rust (via PyO3), and external REST services with sub-100ns invocation overhead for native languages.
    
    \item \textbf{GIL-Free Parallelism:} A multiprocessing worker pool with intelligent DAG affinity scheduling that enables true CPU parallelism by distributing workloads across independent Python interpreters.
\end{enumerate}

\section{Background and Motivation}

\subsection{Dataflow Processing Paradigm}

Dataflow processing systems model computation as directed acyclic graphs (DAGs) where nodes represent computational operations and edges represent data dependencies \cite{johnston2004advances}. This paradigm offers several advantages over imperative programming models:

\begin{itemize}
    \item \textbf{Explicit Parallelism:} Independent nodes can execute concurrently without explicit synchronization
    \item \textbf{Natural Pipeline Expression:} Data transformations are composed declaratively
    \item \textbf{Clear Separation of Concerns:} Computation logic is decoupled from orchestration
    \item \textbf{Fault Isolation:} Node failures can be isolated and recovered independently
\end{itemize}

\subsection{The Serialization Bottleneck}

Inter-process communication traditionally relies on serialization formats such as JSON, Protocol Buffers, or Apache Avro \cite{kleppmann2017designing}. While these formats provide language interoperability, they introduce significant overhead:

\begin{table}[H]
\centering
\caption{Serialization overhead by payload size}
\begin{tabular}{@{}lccc@{}}
\toprule
\textbf{Operation} & \textbf{1KB} & \textbf{100KB} & \textbf{1MB} \\
\midrule
JSON Serialize & 15 $\mu$s & 1.5 ms & 15 ms \\
JSON Deserialize & 20 $\mu$s & 2.0 ms & 20 ms \\
Network Transfer & 5 $\mu$s & 50 $\mu$s & 500 $\mu$s \\
\textbf{Total Overhead} & \textbf{40 $\mu$s} & \textbf{3.5 ms} & \textbf{35 ms} \\
\bottomrule
\end{tabular}
\end{table}

For a 1MB payload, serialization and deserialization can consume 30-50ms of wall-clock time, dominating the total latency budget in microsecond-sensitive applications.

\subsection{Python GIL Constraints}

Python's Global Interpreter Lock (GIL) is a mutex that protects access to Python objects, preventing multiple native threads from executing Python bytecode simultaneously \cite{beazley2010gil}. This design choice simplifies memory management but creates a fundamental throughput limitation: only one thread can execute Python bytecode at any given time, regardless of available CPU cores.

DishtaYantra addresses this through multiprocessing, where each worker runs in a separate Python interpreter with its own GIL.

\subsection{Language Heterogeneity Challenge}

Modern data processing pipelines often require components written in different languages:

\begin{itemize}
    \item \textbf{Python:} Machine learning models (TensorFlow, PyTorch), data science (Pandas, NumPy)
    \item \textbf{Java:} Enterprise systems, Kafka integration, existing business logic
    \item \textbf{C++:} High-performance numerical computing, low-level optimizations, SIMD
    \item \textbf{Rust:} Memory-safe systems programming, concurrent algorithms
    \item \textbf{External Services:} REST APIs, microservices, legacy systems
\end{itemize}

Existing frameworks typically support only one or two languages, forcing developers into suboptimal architectural choices.

\section{System Architecture}

DishtaYantra is architected as a layered system with clear separation of concerns between presentation, orchestration, computation, and infrastructure layers.

\subsection{High-Level Architecture}

The system consists of six architectural layers:

\begin{enumerate}
    \item \textbf{Presentation Layer:} Web UI Dashboard, REST API, Admin Dashboard, Prometheus metrics
    \item \textbf{Application Layer:} Flask application with routing, authentication, and session management
    \item \textbf{Orchestration Layer:} Worker Pool Manager, DAG Affinity Manager, Health Monitor, Auto-Recovery
    \item \textbf{Computation Layer:} ComputeGraph Engine, Node Executor, Time Windows, Subgraph Support
    \item \textbf{Integration Layer:} Multi-language Calculator Framework (Python, Java, C++, Rust, REST)
    \item \textbf{Infrastructure Layer:} LMDB Zero-Copy Transport, Pub/Sub backends (Kafka, RabbitMQ, Redis, etc.)
\end{enumerate}

\subsection{Core Abstractions}

\subsubsection{ComputeGraph}

The primary container representing a complete DAG with the following attributes:

\begin{itemize}
    \item \texttt{name}: Unique identifier
    \item \texttt{nodes}: Dictionary of all nodes in the graph
    \item \texttt{edges}: List of data flow connections
    \item \texttt{status}: STOPPED, RUNNING, SUSPENDED, or ERROR
    \item \texttt{start\_time}: Scheduled start time (e.g., ``0800'')
    \item \texttt{duration}: Run duration (e.g., ``12h'')
    \item \texttt{auto\_clone}: Whether to clone for next day automatically
\end{itemize}

\subsection{Backpressure Management}

DishtaYantra implements comprehensive backpressure to prevent memory exhaustion:

\begin{itemize}
    \item \textbf{BLOCK:} Upstream blocks when queue full (default)
    \item \textbf{DROP:} Excess messages discarded with logging
    \item \textbf{SAMPLE:} Statistical sampling under high load
    \item \textbf{THROTTLE:} Rate limiting at source nodes
\end{itemize}

\subsection{Internal Cache for Crash Recovery}

DishtaYantra maintains an internal cache enabling recovery without data loss. Cache contents include:

\begin{itemize}
    \item In-flight messages currently being processed
    \item Node state for stateful computations
    \item Processing offsets for replay
    \item Checkpoint timestamps for last successful commit
\end{itemize}

Recovery follows at-least-once semantics: load checkpoint, identify in-flight messages, replay from recovery point, and resume normal processing.

  ],

  ]
}
\end{lstlisting}

\subsection{DAG Execution Algorithm}

The execution engine uses Kahn's algorithm for topological sorting:

\begin{enumerate}
    \item Compute topological order of nodes
    \item Initialize input queues for each node
    \item While graph is RUNNING:
    \begin{enumerate}
        \item For each node in execution order:
        \item If node has new input or is marked dirty:
        \item Collect inputs and process
        \item Propagate output to downstream nodes
        \item Mark downstream nodes as dirty
    \end{enumerate}
    \item Sleep for poll interval
\end{enumerate}

\section{Multi-Language Calculator Framework}

\subsection{Calculator Factory Architecture}

The CalculatorFactory uses a type-based dispatch pattern:

\begin{itemize}
    \item \texttt{type: "java"} $\rightarrow$ JavaCalculator (Py4J bridge, $\sim$1ms overhead)
    \item \texttt{type: "cpp"} $\rightarrow$ CppCalculator (pybind11, $\sim$100ns overhead)
    \item \texttt{type: "rust"} $\rightarrow$ RustCalculator (PyO3, $\sim$100ns overhead)
    \item \texttt{type: "rest"} $\rightarrow$ RestCalculator (HTTP, network-dependent)
    \item \texttt{type: "python"} or default $\rightarrow$ Built-in Python calculator
\end{itemize}

\subsection{Java Integration (Py4J)}

Java calculators communicate via Py4J TCP bridge (port 25333). A connection pool manages gateway connections. Performance is approximately 1ms per call due to TCP overhead.

\subsection{C++ Integration (pybind11)}

C++ calculators are compiled as Python extension modules (\texttt{.so} files) using pybind11 \cite{pybind11}. This provides direct memory access with approximately 100ns call overhead.

\subsection{Rust Integration (PyO3)}

Rust calculators use PyO3 \cite{pyo3} for Python bindings, offering memory safety, thread safety, and zero-cost abstractions with similar performance to C++.

\subsection{REST Integration}

REST calculators enable integration with external microservices via HTTP POST requests. Supports multiple authentication methods: API key, basic, bearer, and custom headers.

\section{LMDB Zero-Copy Data Exchange}

\subsection{Design Principles}

Traditional inter-process communication requires serialization at the sender and deserialization at the receiver. DishtaYantra's LMDB-based approach eliminates this overhead by leveraging memory-mapped files.

\textbf{Key insight:} LMDB \cite{chu2011lmdb} provides a lightning-fast, memory-mapped key-value store with ACID guarantees. By writing data once to LMDB and passing only the key reference to native calculators, we achieve zero-copy reads across process and language boundaries.

\subsection{Exchange Protocol}

\begin{enumerate}
    \item \textbf{Step 1:} Python DAG engine writes input data to LMDB with a unique key
    \item \textbf{Step 2:} Native calculator (Java/C++/Rust) reads directly from memory-mapped file using the key
    \item \textbf{Step 3:} Calculator writes output back to LMDB; Python reads result
\end{enumerate}

\section{Worker Pool Architecture}

\subsection{Overview}

DishtaYantra's worker pool enables true CPU parallelism by running DAGs in separate Python interpreter processes, each with its own GIL.

The main process contains:
\begin{itemize}
    \item WorkerPoolManager for process lifecycle management
    \item DAG Affinity Manager for intelligent scheduling
    \item Health Monitor for failure detection
    \item Auto-Recovery for crashed worker restart
\end{itemize}

Workers communicate via LMDB pub/sub for cross-worker IPC.

\subsection{DAG Affinity Scheduling}

\midrule
\texttt{weight\_based} & Assigns to lowest-load worker & General use \\

\subsection{Fault Tolerance}

The health monitor runs every 2 seconds. If a worker misses 3 consecutive heartbeats:
\begin{enumerate}
    \item Worker marked as UNHEALTHY
    \item DAGs migrated to healthy workers
    \item Restart attempted with exponential backoff (5s, 10s, ..., max 60s)
\end{enumerate}

\section{Related Work}

\subsection{Stream Processing Frameworks}

\textbf{Apache Flink} \cite{carbone2015flink} provides exactly-once stream processing with sophisticated windowing semantics. However, Flink's Python API (PyFlink) introduces significant overhead due to cross-language serialization.

\textbf{Apache Spark} \cite{zaharia2016spark} pioneered in-memory distributed computing with the RDD abstraction. Spark Streaming processes data in micro-batches, introducing minimum latencies of 100ms-1s.

\textbf{Naiad} \cite{murray2013naiad} introduced timely dataflow for iterative computations with low-latency incremental updates.

\subsection{Zero-Copy Technologies}

\textbf{Apache Arrow} \cite{arrow2016} provides a columnar memory format enabling zero-copy data sharing. DishtaYantra extends this concept to arbitrary data structures via LMDB's memory-mapped B-tree.

\subsection{Workflow Orchestration}

\textbf{Prefect} \cite{prefect2018} and \textbf{Dagster} \cite{dagster2019} are modern Python-native workflow orchestration tools, but lack DishtaYantra's focus on microsecond-latency processing.

\section{Discussion}

\subsection{Design Trade-offs}

\textbf{Memory vs. Latency:} LMDB's memory-mapped design trades memory usage for latency. The default 1GB map size accommodates most workloads.

\textbf{Process Isolation vs. Communication Overhead:} The multiprocessing architecture eliminates GIL contention but introduces IPC overhead. LMDB pub/sub minimizes this cost.

\textbf{Flexibility vs. Complexity:} Supporting five languages increases implementation complexity but enables optimal algorithm selection.

\subsection{Limitations}

\begin{enumerate}
    \item \textbf{Single-Node Focus:} Current implementation targets single-node deployments
    \item \textbf{JVM Startup Time:} Java calculators require JVM initialization ($\sim$2-5 seconds)
    \item \textbf{Memory Footprint:} Each worker process maintains independent memory
\end{enumerate}

\subsection{Future Work}

\begin{enumerate}
    \item \textbf{Distributed Mode:} Multi-node deployment with automatic DAG partitioning
    \item \textbf{GPU Calculators:} CUDA/ROCm integration for ML inference
    \item \textbf{Checkpoint/Recovery:} Exactly-once semantics with distributed checkpointing
    \item \textbf{WebAssembly Calculators:} Sandboxed calculator execution via WASM
\end{enumerate}

\section{Conclusion}

We presented \textbf{DishtaYantra}, a high-performance multi-language dataflow framework that addresses critical limitations in existing systems. Through LMDB-based zero-copy data exchange, unified multi-language calculator support, and GIL-free worker parallelism, DishtaYantra achieves:

\begin{itemize}
    \item \textbf{100-250$\times$ latency improvements} over traditional JSON serialization for large payloads
    \item \textbf{Sub-millisecond P99 latency} for payloads up to 100KB
    \item \textbf{90\% parallel efficiency} at 12 workers on consumer hardware
    \item \textbf{Unified support} for Python, Java, C++, Rust, and REST calculators
    \item \textbf{Comprehensive pub/sub} integration with 10+ message brokers
\end{itemize}

DishtaYantra is open-source and available at \url{https://github.com/ajsinha/dishtayantra}.
