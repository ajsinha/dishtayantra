User Database (v2.0.0)

Users, roles, and API keys now live in a relational database (SQLite by default, PostgreSQL by configuration) with PBKDF2-hashed passwords - replacing the plaintext users.json file.

Overview
  • SQLAlchemy + DAO layer - users, roles, user_roles, and api_keys tables
  • Hashed credentials - PBKDF2-SHA256 with per-user salt; clear passwords are never stored or exposed by any API
  • API keys - issued per user (dyk_...), shown once, stored hashed, with optional expiry, revocation, and last-used tracking
  • Safety rails preserved - the last remaining admin can be neither deleted nor stripped of the admin role
One-time migration: on first start, an existing config/users.json is imported into the database (passwords are hashed during import) and the file is renamed users.json.migrated so the import never repeats. With no legacy file and an empty database, a bootstrap admin / admin123 account is created - change its password immediately.
Configuration
SQLite (default)
db.engine=sqlite db.sqlite.path=data/dishtayantra.db

The schema is created automatically on first start from config/schema/schema_sqlite.sql. There are no migrations to run.

PostgreSQL
db.engine=postgresql db.postgres.host=localhost db.postgres.port=5432 db.postgres.database=dishtayantra db.postgres.user=${POSTGRES_USER:dishtayantra} db.postgres.password=${POSTGRES_PASSWORD:}

Apply config/schema/schema_postgres.sql once before the first start (psql -f config/schema/schema_postgres.sql). The server validates connectivity at startup and fails fast with a detailed error if any db.* property is missing - nothing is defaulted silently.

API keys

Manage keys from the User Management page or the /users/api/... endpoints. The clear key is displayed exactly once at creation:

dyk_Z8f3kQ... # save it now - only its hash is stored

Revoked or expired keys are rejected immediately; last_used_at is stamped on every successful verification.

Security notes
  • Set a real session secret: app.secret_key (or the SECRET_KEY env var). The server refuses to start without one.
  • Database files under data/ should be excluded from backups containing secrets only if your policy requires it - they contain hashes, not passwords.
  • All database errors surface with full stack traces in the logs; UI operations show a flash message and log the trace.
