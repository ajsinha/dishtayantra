Worker Pool & DAG Affinity

Multiprocessing for True CPU Parallelism

v1.5.2
Back to Help Open Worker Pool UI
Overview

DishtaYantra v1.5.2 introduces a multiprocessing worker pool that enables true CPU parallelism by distributing DAGs across multiple worker processes.

Key Benefits
  • True Parallelism - Bypass Python's GIL
  • DAG Affinity - Keep related DAGs together
  • Cache Locality - Better performance
  • Fault Isolation - Worker crash doesn't affect others
  • Auto-Restart - Automatic recovery
Components
  • WorkerPoolManager - Orchestrates workers
  • DAGWorkerProcess - Individual worker
  • AffinityManager - DAG assignment
  • HealthMonitor - Crash detection
  • LMDB Pub/Sub - Cross-worker communication

Configuration
Worker Pool Configuration File

All worker pool settings are managed via config/worker_config.json:

{
    "worker_pool": {
        "enabled": false,  // Set to true to enable multiprocessing
        "num_workers": 4,  // Default is 4 (use "auto" for CPU count)
        "min_workers": 2,
        "max_workers": 32,
        "auto_restart_on_crash": true,
        "max_restart_attempts": 3,
        "restart_backoff_seconds": 5,
        "health_check_interval_seconds": 5
    },
    
    "affinity": {
        "default_strategy": "weight_based",
        "rebalance_enabled": true,
        "rebalance_threshold": 0.3,
        "allow_dag_pinning": true,
        "allow_exclusive_workers": true
    },
    
    "communication": {
        "use_lmdb_for_cross_worker": true,
        "lmdb_path": "data/worker_lmdb",
        "lmdb_map_size_mb": 1024
    },
    
    "monitoring": {
        "collect_cpu_metrics": true,
        "collect_memory_metrics": true
    }
}
Worker Pool Options
Option Type Default Description
enabled boolean true Enable/disable worker pool
num_workers int/string 4 Number of workers (use "auto" for CPU count)
min_workers int 2 Minimum workers when using "auto"
max_workers int 32 Maximum workers when using "auto"
auto_restart_on_crash boolean true Auto-restart crashed workers
max_restart_attempts int 3 Max consecutive restart attempts
restart_backoff_seconds int 5 Initial backoff (exponential)
Affinity Options
Option Type Default Description
default_strategy string "weight_based" DAG assignment strategy
rebalance_enabled boolean true Enable automatic load rebalancing
rebalance_threshold float 0.3 Load difference to trigger rebalance
allow_dag_pinning boolean true Allow DAGs to pin to specific workers
DAG Worker Affinity

Control which worker a DAG is assigned to by adding worker_affinity to your DAG configuration:

Pin to Specific Worker
{
    "name": "critical_trading_dag",
    "worker_affinity": {
        "pinned_worker": 0
    }
}
Exclusive Worker
{
    "name": "high_frequency_dag",
    "worker_affinity": {
        "exclusive": true
    }
}
Preferred Workers
{
    "name": "analytics_dag",
    "worker_affinity": {
        "preferred_workers": [1, 2]
    }
}
Tip: Pin related DAGs to the same worker to benefit from shared calculator caches and reduce cross-process communication.
Cross-Worker Communication with LMDB

Use LMDB-based pub/sub for efficient cross-worker data sharing:

Publisher (Worker 0)
{
    "name": "cross_worker_pub",
    "config": {
        "destination": "lmdb://shared_channel"
    }
}
Subscriber (Worker 1)
{
    "name": "cross_worker_sub",
    "config": {
        "source": "lmdb://shared_channel"
    }
}
Why LMDB?
  • Memory-mapped - Zero-copy reads across processes
  • Multiple readers - Many workers can read simultaneously
  • ACID compliant - Data integrity guaranteed
  • Persistent - Survives worker restarts
Health Monitoring & Auto-Restart
How It Works
  1. Heartbeat Monitoring - Workers send heartbeats every 2 seconds
  2. Unhealthy Detection - No heartbeat for 3 intervals = unhealthy
  3. Crash Detection - Process not alive = crashed
  4. Auto-Restart - Crashed workers restart with exponential backoff
  5. DAG Restoration - DAGs automatically reloaded on restarted worker

Best Practices
DO
  • Co-locate related DAGs on the same worker
  • Use exclusive workers for critical DAGs
  • Use LMDB for cross-worker communication
  • Monitor worker health regularly
  • Start with 4 workers (default), tune later
DON'T
  • Don't create more workers than CPU cores
  • Don't pin all DAGs (let affinity work)
  • Don't ignore high restart counts
  • Don't use in-memory queues across workers
