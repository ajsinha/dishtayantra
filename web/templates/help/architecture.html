System Overview

DishtaYantra is a high-performance, multi-threaded DAG (Directed Acyclic Graph) compute server designed for real-time data processing pipelines.

DAG Processing
Parallel node execution with topological sorting
Multi-Broker
Kafka, RabbitMQ, Redis, LMDB, and more
Multi-Language
Python, Java, C++, Rust, REST calculators
Worker Pool
Multiprocess parallelism bypassing GIL
LMDB PubSub
Zero-copy cross-process messaging
High Availability
Zookeeper-based leader election

Subgraph Architecture (Graph-as-a-Node)

Subgraphs enable modular, reusable computation units that can be dynamically controlled at runtime.

Key Concept: A subgraph is a complete computation graph that appears as a single node in the parent graph, enabling hierarchical composition and dynamic "light up/light down" control.

Light Up / Light Down Control
рҹҹў Light Up (Active)
  • Subgraph participates in calculation cycles
  • Dirty signals propagate through internal nodes
  • Fresh calculations are performed
рҹ”ҙ Light Down (Suspended)
  • Subgraph is skipped in calculation cycles
  • Dirty signals STOP at subgraph boundary
  • Cached output is returned (configurable)
Dark Output Modes
Mode Behavior When Suspended Use Case
cached Returns last computed output Most common - maintains downstream consistency
passthrough Passes input directly to output When subgraph is optional/bypass-able
block Returns None, blocks downstream When downstream should not execute

Key Components
LMDBTransport
  • Memory-mapped file operations
  • Multiple data formats (JSON, MsgPack, NumPy, Arrow)
  • Automatic compression for large payloads
  • TTL-based expiration and cleanup
  • ACID transaction guarantees

  • Automatic payload size detection
  • Unified reference protocol for all languages
  • Transaction-based exchange with cleanup
  • Configurable thresholds (default 1KB)
  • Path configured in application.properties

LMDB PubSub for Cross-Worker Communication

Workers communicate via LMDB-based pub/sub - a memory-mapped, zero-copy transport ideal for inter-process communication:

  • Memory-Mapped I/O - Direct memory access, no serialization overhead
  • Multiple Readers - Many workers can read simultaneously
  • ACID Compliant - Data integrity guaranteed
  • Persistent - Data survives worker restarts
URI Format: lmdb://channel_name
DAGs on different workers can publish/subscribe to shared LMDB channels.
