High Availability Manager (v2.0.0)

Run multiple DishtaYantra instances with automatic primary election and failover - the coordination backend is chosen by configuration.

Overview

The HA Manager elects exactly one primary instance; all others stand by as secondaries. On election the new primary resumes the DAGs that were suspended; on demotion the instance suspends its running DAGs. Five providers are available via ha.provider:

  • none (default) - standalone; this instance is always primary
  • zookeeper - leader election through an ephemeral-sequential znode (kazoo)
  • redis - expiring lock key acting as a renewable lease
  • s3 - lease object in a bucket with timestamp-based takeover
  • socket - exclusive TCP port bind; whoever holds the port is primary
Configuration
Standalone (default)
ha.provider=none
Zookeeper
ha.provider=zookeeper ha.zookeeper.hosts=zk1:2181,zk2:2181,zk3:2181 ha.zookeeper.election_path=/dishtayantra/election
Redis lease
ha.provider=redis ha.redis.host=localhost ha.redis.port=6379 ha.redis.lock_key=dishtayantra:ha:lock ha.redis.lease_seconds=10
S3 lease
ha.provider=s3 ha.s3.bucket=my-dishtayantra-bucket ha.s3.lease_key=dishtayantra/ha/lease.json ha.s3.region=us-east-1 ha.s3.lease_seconds=15
Socket (exclusive bind)
ha.provider=socket ha.socket.host=0.0.0.0 ha.socket.port=5599 ha.socket.retry_seconds=3
Failover behaviour
  • Elected → the instance becomes primary and resumes the DAGs it suspended while standing by.
  • Demoted → the instance suspends all running DAGs and waits to be re-elected.
  • Connecting to the socket provider's port returns a one-line status banner, handy for probes:
$ nc primary-host 5599 DishtaYantra v2.0.0 HA primary on prod-node-1
Pair HA with a shared storage provider (S3/Azure/GCS - see Storage Abstraction) so every instance sees the same DAG definitions.
Choosing a provider
  • zookeeper - strongest guarantees; pick when a ZK ensemble already exists.
  • redis - lightweight and fast when Redis is already deployed.
  • s3 - no extra infrastructure when running in AWS; slightly slower failover.
  • socket - zero dependencies for two instances on one network; the legacy "spocket" mechanism formalized.
