Processing Interrelated Batch Files (End-of-Day Feeds)

How to use DishtaYantra when data arrives as large, interrelated batch files (e.g. an EOD trade feed enriched against EOD FX-rate and client-limit feeds) rather than as a live stream.

The short answer

Yes — this is a great fit, and usually you do not need one-record-at-a-time. A batch file is just a bounded stream: read it and replay its records onto the DAG's input queue (or point a file / object-store subscriber at it). The only real design question is how your feeds are interrelated.

Runnable demo: python -m perftest.run_eod_example

Two kinds of "interrelated"

(a) Fact × Dimension — the common case, NOT one-at-a-time. Trades are the large fact feed; FX rates and client limits are small dimension feeds. A trade needs its FX rate (by currency) and its limit (by client), but trades do not depend on each other. Load the small dimension feeds once into in-memory lookup tables, then stream the trade feed through, enriching each trade by lookup. Independent per trade → fast, parallel, vectorizable. No ordering needed.

(b) Record × Record — the genuine one-at-a-time case. If a record depends on the records before it (e.g. a client's running exposure accumulates trade by trade), records must be processed in order with state carried forward. This is the only case that truly needs sequential processing — handled with a stateful calculator on an ordered stream.

Architecture
            EOD FX-rate feed  ─┐  loaded ONCE into
            EOD limits feed   ─┘  in-memory lookup tables
                               │
EOD trade file  →  [replay onto queue]  →  DAG: enrich + value + limit-check  →  output
   (fact)              (bounded stream)        (FX lookup, notional, breach)

trades = fact feed (large, replayed); fx_rates + client_limits = dimension feeds (small, loaded once).

Load the dimension feeds once

A calculator loads reference tables in its constructor — inline or from a file path:

{
  "name": "enrich",
  "type": "perftest.eod_enrichment_calculators.ReferenceEnrichCalculator",
  "config": {
    "fx_file": "/feeds/eod/fx_rates.json",
    "limits_file": "/feeds/eod/client_limits.json",
    "default_rate": 1.0
  }
}

For very large or shared reference data, load it into the LMDB store or the Redis-clone instead of a per-calculator dict; the lookup pattern is identical.

Replay the trade file, enrich, and check

Publish each file record onto the input queue (in production, point a file / Kafka / object-store subscriber at the feed instead — the DAG is identical):

import json
from core.pubsub.inmemorypubsub import InMemoryPubSub
ps = InMemoryPubSub()
with open("/feeds/eod/trades.jsonl") as fh:
    for line in fh:
        ps.publish_to_queue("eod_trades_in", line.strip())

The DAG is ingest → enrich → sink. The enricher looks up FX and limit, computes notional, and flags breaches (output adds fx_rate, notional_usd, client_limit, limit_breach).

Making it fast
  • Dimensions in memory → O(1) lookups, no per-record I/O.
  • Auto-batching + Arrow → put a BatchingSubscriptionNode at the source and use Arrow (vectorized) calculators, or wrap a row enricher with RowCalculatorBatchAdapter, then a FlatteningPublicationNode so output stays per-record. See the Arrow tutorial (docs/TUTORIAL_arrow.md).
  • Parallelism → partition the trade file (by symbol or client) and run multiple DAG instances / the worker pool; each partition shares the same loaded dimension tables.
The sequential case (true one-at-a-time)

For running exposure, process the ordered trade stream through a stateful calculator (RunningExposureCalculator) that keeps a per-client cumulative notional and flags the trade that breaches the limit. Keys to correctness:

  • Order matters — feed records in the intended order; don't shuffle.
  • Keep the dependency on one partition — shard by client across workers (each client's trades stay ordered within its shard), but don't vectorize across the running dependency.
  • Enrich first, then accumulate — do the parallel enrichment (case a) up front, then the sequential step (case b) on the result.
Runnable example
python -m perftest.run_eod_example --trades 20000
  • perftest/eod_enrichment_calculators.py — ReferenceEnrichCalculator (fact × dimension) and RunningExposureCalculator (sequential).
  • perftest/run_eod_example.py — loads FX + limits, replays the trade file, runs enrichment one-at-a-time and auto-batched, and reports breaches + throughput.
