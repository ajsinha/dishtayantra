Storage Abstraction (v2.0.0)

Store DAG configurations on the local filesystem, Amazon S3, Azure Blob Storage, or Google Cloud Storage - switchable purely through configuration.

Overview

In v2.0.0 the server never touches DAG configuration files directly. All reads and writes flow through a StorageProvider selected by storage.provider in config/application.properties:

  • filesystem (default) - local directory tree, drop-in compatible with v1.x layouts
  • s3 - Amazon S3 or any S3-compatible store (MinIO, LocalStack) via boto3
  • azureblob - Azure Blob Storage via azure-storage-blob
  • gcs - Google Cloud Storage via google-cloud-storage

The logical location of DAG configs is storage.dags.prefix (default config/dags). Only direct children of this prefix are auto-loaded - sub-prefixes such as config/dags/examples/ hold samples that are not started automatically.

Missing configuration is a fatal error: the server refuses to start and logs exactly which property is absent. Nothing is silently defaulted.
Configuration
Filesystem (default)
storage.provider=filesystem storage.filesystem.root=./ storage.dags.prefix=config/dags
Amazon S3
storage.provider=s3 storage.s3.bucket=my-dishtayantra-bucket storage.s3.prefix=dishtayantra storage.s3.region=us-east-1 # Optional - custom endpoint (MinIO / LocalStack) and explicit credentials: #storage.s3.endpoint_url=http://localhost:9000 #storage.s3.access_key_id=${AWS_ACCESS_KEY_ID:} #storage.s3.secret_access_key=${AWS_SECRET_ACCESS_KEY:}
Azure Blob Storage
storage.provider=azureblob storage.azureblob.connection_string=${AZURE_STORAGE_CONNECTION_STRING:} storage.azureblob.container=dishtayantra storage.azureblob.prefix=dishtayantra
Google Cloud Storage
storage.provider=gcs storage.gcs.bucket=my-dishtayantra-bucket storage.gcs.prefix=dishtayantra storage.gcs.credentials_file=/etc/gcp/service-account.json

Install the matching optional dependency: boto3, azure-storage-blob, or google-cloud-storage (see requirements.txt).

What flows through storage
  • Loading all DAG configurations at startup
  • Saving new DAGs created in the DAG Designer or via the API
  • Reading DAG configs for clone / subgraph resolution
Operational notes
  • With a cloud provider, multiple server instances share one set of DAG definitions - pair this with the HA Manager for active/standby deployments.
  • Paths are validated - traversal outside the configured root/prefix is rejected.
  • All storage failures are logged with full stack traces; nothing is swallowed.
