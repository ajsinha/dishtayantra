Headless Execution & Orchestration

Run ETL without the web UI, and let a long-running control-plane instance dispatch ephemeral headless workers for heavy, run-to-completion jobs.

Headless run-once (the worker)

The engine never needed the UI — the UI is a monitor/designer. The headless runner starts a DAG, processes a bounded feed to completion, drains, stops cleanly, writes a summary, and exits with a status code:

python -m core.dag.headless_runner --config job.json \
    [--replay feed.jsonl] [--expect N] [--summary out.json] [--job-id ID]

Completion is explicit: count-based (sink published reaches the expected count) or quiescence (no dirty nodes and empty input queues for several polls). Shutdown drains first, honouring the zero-message-loss invariant. A bounded file is just a finite stream, so this is the natural way to run an EOD job.

Orchestration — control plane + workers

A long-running control-plane instance (with the UI, for observability) reacts to events and launches ephemeral headless workers for heavy ETL. This separates an always-on, light decision plane from isolated, disposable compute — a crashing job can't take down the control plane, and finished workers just exit.

job_requests -> [dispatch] -> sink          (launch headless workers)
job_done     -> [react]    -> ...           (react to completions)

A JobDispatchCalculator node turns qualifying events into worker processes.

The dispatcher's guarantees
  • Idempotent — exactly one launch per unique job key, even on recompute or a replayed event (a dedup set complements the equality gate).
  • Asynchronous — launches and returns immediately; a daemon thread waits on each child, so the reactive loop never blocks.
  • Bounded — a max_concurrent cap with a pending queue (a real worker pool), so an event burst cannot fork-bomb the host.
  • Observable — on child exit it records the summary and can publish a completion event the parent subscribes to (reactive loop). shutdown() terminates live children so the control plane never orphans them.
Choosing the shape
  • External scheduler launches a run-once worker (Autosys/Control-M/Airflow watches for the feed, then runs the worker). Leanest and most standard.
  • In-DAG sensing + dispatch — a control-plane DAG senses readiness (a metronome polling a manifest/.done/control total) and dispatches workers itself. Self-contained.
  • Process isolation vs in-process parallelism — separate headless processes give isolation, ephemerality, and run-to-completion; for pure throughput the in-process worker pool / autoclone is lighter.

Prefer an explicit completeness signal (manifest, trailer record, control count) over time-based heuristics.
