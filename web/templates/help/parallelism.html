Choosing a Parallelism Strategy

DishtaYantra v1.5.2 offers two complementary approaches to achieve parallelism and maximize CPU utilization:

Worker Pool (Multiprocessing)

Distributes DAGs across multiple processes. Each worker runs in its own Python interpreter, completely bypassing the GIL.

  • Best for: Production environments
  • Stability: Mature, battle-tested
  • Isolation: Worker crashes don't affect others
Free-Threading (No-GIL)

Uses Python 3.13+ with GIL disabled for true thread-level parallelism within a single process.

  • Best for: Experimentation, specific workloads
  • Stability: Experimental (Python 3.13+)
  • Memory: Shared memory between threads

Quick Start
Step 1: Enable Worker Pool

Edit config/worker_config.json:

{
    "worker_pool": {
        "enabled": true,  // Change from false to true
        "num_workers": 4     // Or "auto" for CPU count
    }
}
Step 2: Start DishtaYantra
python run_server.py
Step 3: Verify Workers

Open Admin → Worker Pool to see all workers running.

Tip: Start with 4 workers (default), then tune based on your workload and available CPU cores.

Cross-Worker Communication (LMDB)

DAGs on different workers can communicate via LMDB pub/sub using the lmdb:// URI scheme:

Publisher (Worker 0)
{
    "name": "cross_worker_pub",
    "config": {
        "destination": "lmdb://shared_channel"
    }
}
Subscriber (Worker 1)
{
    "name": "cross_worker_sub",
    "config": {
        "source": "lmdb://shared_channel"
    }
}
Why LMDB?
  • Memory-mapped - Zero-copy reads across processes
  • Multiple readers - Many workers can read simultaneously
  • ACID compliant - Data integrity guaranteed
  • Persistent - Survives worker restarts
Health Monitoring & Auto-Restart
How It Works
  1. Heartbeat Monitoring - Workers send heartbeats every 2 seconds
  2. Unhealthy Detection - No heartbeat for 3 intervals = unhealthy
  3. Crash Detection - Process not alive = crashed
  4. Auto-Restart - Crashed workers restart with exponential backoff
  5. DAG Restoration - DAGs automatically reloaded on restarted worker

Best Practices
DO
  • Co-locate related DAGs on the same worker
  • Use exclusive workers for critical DAGs
  • Use LMDB for cross-worker communication
  • Monitor worker health regularly
  • Start with 4 workers (default), tune later
DON'T
  • Don't create more workers than CPU cores
  • Don't pin all DAGs (let affinity work)
  • Don't ignore high restart counts
  • Don't use in-memory queues across workers
What is Free-Threading?

Python has historically had a Global Interpreter Lock (GIL) - a mutex that protects access to Python objects, preventing multiple threads from executing Python bytecode simultaneously. This means that even on a 32-core machine, only one thread could execute Python code at a time.

Traditional Python (with GIL)
Thread 1 (Running) Thread 2 (Blocked) Thread 3 (Blocked) Thread 4 (Blocked)

Only one thread executes Python code at a time

Free-Threading (also called "no-GIL" or "GIL-free") removes this limitation, allowing multiple threads to execute Python code truly in parallel across all CPU cores.

Free-Threading Python (no GIL)
Thread 1 (Running) Thread 2 (Running) Thread 3 (Running) Thread 4 (Running)

All threads execute Python code simultaneously

Timeline:
  • Python 3.13 (October 2024): Experimental free-threading support introduced
  • Python 3.14 (October 2025): Improved free-threading with better performance
  • Future: Free-threading expected to become the default
Why Free-Threading Matters for DishtaYantra

DishtaYantra is inherently a multi-threaded application. Each DAG creates multiple threads:

Component Threads Per Instance Purpose
Subscribers1 per subscriberListen for incoming messages
Publishers1 per publisher (if batched)Send outgoing messages
Compute Thread1 per DAGProcess data through nodes
Time Window Checker1 per DAG (if configured)Monitor time windows

Potential Performance Impact
Compute-bound workloads Up to 10-20x faster
Mixed workloads 2-5x faster
I/O-bound workloads Minimal change (already parallelized)
Quick Start: Building Free-Threading Python
Option 1: Build from Source
# Download Python 3.13+
wget https://www.python.org/ftp/python/3.13.0/Python-3.13.0.tgz
tar xzf Python-3.13.0.tgz
cd Python-3.13.0

# Configure with --disable-gil
./configure --disable-gil --enable-optimizations --prefix=/opt/python-3.13-freethreaded

# Build and install
make -j$(nproc)
sudo make install

# Create virtual environment
/opt/python-3.13-freethreaded/bin/python3 -m venv venv-ft
source venv-ft/bin/activate

# Verify
python --version
python -c "import sys; print(f'GIL: {sys._is_gil_enabled()}')"

Thread Safety in Custom Calculators

With free-threading, your custom calculators may run truly in parallel. Ensure thread safety:

import threading

class MyCalculator(DataCalculator):
    def __init__(self, name, config):
        super().__init__(name, config)
        self._lock = threading.Lock()  # Use locks for shared state
        self._counter = 0
    
    def calculate(self, data):
        # Thread-safe counter increment
        with self._lock:
            self._counter += 1
            local_count = self._counter
        
        # Process data (this part runs in parallel)
        result = self._process(data)
        return result
Avoid Global Mutable State
# BAD - Global mutable state
results = []  # This will have race conditions!

class BadCalculator(DataCalculator):
    def calculate(self, data):
        results.append(data)  # Race condition!
        return data

# GOOD - Instance-level or thread-local state
class GoodCalculator(DataCalculator):
    def __init__(self, name, config):
        super().__init__(name, config)
        self._results = []
        self._lock = threading.Lock()
    
    def calculate(self, data):
        with self._lock:
            self._results.append(data)
        return data
Warning: Free-threading is still experimental. Some packages may not yet support it. Check for compatible versions.
Troubleshooting
Issue: Package Installation Fails
# Some C extensions may not support free-threading yet
# Try installing without binary wheels
pip install package-name --no-binary :all:

# Or check for a compatible version
pip install package-name==version
Issue: Performance Not Improving
  • Verify GIL is actually disabled: sys._is_gil_enabled()
  • Check if workload is I/O bound (I/O bound tasks already benefit from threading)
  • Ensure you have enough CPU cores for parallel execution
  • Profile your code to find bottlenecks
Issue: Segmentation Faults
# Identify problematic packages
python -c "import package_name"  # Test each package

# Run with debugging
PYTHONFAULTHANDLER=1 python run_server.py
