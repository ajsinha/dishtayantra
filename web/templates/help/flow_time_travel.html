Flow Time-Travel

Record the engine's change-log and replay any DAG's history — what each node output, when, and what the graph looked like at a chosen moment.

Overview

Because nodes only recompute when an input actually changes (the equality gate), the flow history is a compact change-log rather than a full event stream. The recorder captures each gate fire off the hot path (a bounded queue feeds a single background writer), so capture never blocks computation.

Open it from the navbar: Manage → Flow Time Travel (or /flow). You can pick a DAG, scrub a time window to see the events, reconstruct the state at a point in time, and export the window as JSONL or CSV.

On by default (SQLite). Set flow_recorder.enabled=false to disable, or an admin can toggle capture on/off at runtime from Admin → System Monitoring — globally (all DAGs) or for a single DAG via the scope dropdown. (The Flow page itself shows recording status read-only.) The same is available from the dyflow CLI: dyflow enable|disable [--dag NAME] (the runtime toggle is in-memory and does not persist across a restart).

Storage backends

The flow store is always a database separate from the application database. Selected by flow_recorder.store:

storeWhat it is
sqlite defaultA separate SQLite file (flow_recorder.store_path, default data/flow_history.db). Frugal, zero extra dependencies.
postgresA dedicated SQLAlchemy database via flow_recorder.db_url. The same code targets PostgreSQL or any URL. Several instances can share one central flow DB.
noopKeeps the wiring but persists nothing.
paimonOptional Apache Paimon append-only archive (pure-Python PyPaimon).
aerospikeOptional, experimental.
daoForbidden. Flow history must never share the application database; the server rejects it.

Every event is stamped with provenance (instance, host, port), so a shared database stays origin-aware: queries can scope to one instance and the DAG inventory groups by origin.

Configuration
flow_recorder.enabled=true # capture on at startup (default ON) flow_recorder.store=sqlite # sqlite | postgres | noop | paimon | aerospike flow_recorder.store_path=data/flow_history.db # store=sqlite (separate file) #flow_recorder.db_url=postgresql+psycopg://user:pass@host:5432/flowdb # store=postgres #flow_recorder.warehouse=data/flow_warehouse # store=paimon flow_recorder.maxsize=100000 # bounded queue; overflow dropped (counted) flow_recorder.max_payload_bytes=2048 # per-side JSON cap on captured values flow_recorder.sample_rate=1.0 # 1.0 = every fire; lower to shed load flow_recorder.retention_hours=24 # history kept; <= 0 disables purging flow_recorder.retention_sweep_minutes=30 flow_recorder.maintenance=none # none | incremental | full (sqlite VACUUM)

See the Configuration Reference Guide for the full key list.

Backend caveats

postgres — implemented as dialect-agnostic SQLAlchemy and exercised via SQLite in tests; validate against a live PostgreSQL before production. Provisioning: SQLite auto-creates the schema, but on PostgreSQL the application does not — apply config/schema/flow_events_postgres.sql once yourself before enabling.

paimon — a real append-only, hourly-partitioned backend. On a local filesystem Paimon catalog, partition-drop retention is a no-op (PyPaimon needs a REST catalog to drop partitions); read/write are unaffected. Bound retention with a REST catalog or the table's partition.expiration-time.

aerospike — complete but experimental and not yet verified against a live cluster.

API & CLI
GET /flow # the Time-Travel console GET /api/flow/status # recorder stats + enabled flag POST /api/flow/enable[?dag=NAME] | /api/flow/disable[?dag=NAME] # admin: toggle (all or one DAG) GET /api/flow/dags # DAGs with event counts + time bounds GET /api/flow/{dag_id}?from&to&nodes&limit&after # events in a window GET /api/flow/{dag_id}/state-at?ts=ms # reconstructed state GET /api/flow/{dag_id}/distribution # rolling 24h cycle distribution (in-memory) GET /api/flow/{dag_id}/histogram?from&to&buckets=120 # exact cycle/event counts (SQL) GET /api/flow/{dag_id}/stream?from&to&cycles=256 # SSE replay, grouped by compute cycle GET /api/flow/{dag_id}/export?from&to&format=jsonl|csv&download=1

Reading the graph: during replay each node displays a running message count beneath its name, so you can watch the DAG light up partially depending on the data and branch logic. The Clear button stops the stream, releases the server slot and resets the graph, counts and event list in one action. The window is chosen with simple Today/Yesterday + hh:mm:ss AM/PM dropdowns (no calendar).

Window & distribution chart: the backend keeps a rolling last-24h distribution per DAG in memory (120 x 12-min buckets), updated as events are recorded. The bottom-pane chart loads this with a single cheap read (/distribution, no DB scan), so many concurrent replays cost the database nothing. On load the window auto-focuses to where the DAG was actually active (so a quiet 24h doesn't look sparse); adjusting the From / To pickers (max 24h) just re-slices the buckets already in the browser - no further server hits. An amber playhead advances during replay. /histogram remains available for an exact SQL aggregate over an arbitrary window; Download exports the window as JSONL.

Large windows & compute cycles: the replay page streams the selected window from the server as Server-Sent-Event pushes and renders them through a bounded queue, so the browser stays responsive even for windows with hundreds of thousands of events. Each push carries a configurable number of complete compute cycles (cycles, default 256) - a compute cycle is every node fire from one engine sweep (one start-to-finish propagation wave), grouped by cycle_id - so the display is never a partial cycle and cycles arrive in order. Pick a range (max 24h) with the From/To controls or the timeline handles, then press play; Download exports the same window as JSONL (one JSON record per line).

Replay limits: concurrent replays are capped server-side (flow_recorder.max_concurrent_streams, default 5) and each replay has a duration cap (flow_recorder.stream_max_seconds, default 120s). The page header shows how many replays are in progress; at capacity, a new replay is asked to try again shortly. The transport bar has a Stop button that disconnects the stream immediately and releases the server slot (the same happens automatically if you pause or leave the page). These guards protect the web tier only - DAG computation is already isolated (separate WAL flow DB; the compute hot path only enqueues).

A command-line client is provided at tools/dyflow.py.

SLO / staleness alerts

Opt-in staleness alerting is derived from the same flow history. A rule breaches when a DAG (or a named node) has produced no output change within its limit. Because of the equality gate this is output-change staleness — a constant-but-healthy stream reads as stale, so size max_age_seconds to the slowest legitimate change interval.

Enable with alerts.enabled=true and point alerts.rules_file at a JSON rules file (see config/alerts.example.json). Status appears on Admin → System Monitoring, at GET /api/alerts, and via dyflow alerts.

{"rules":[ {"name":"trade-etl alive","dag":"trade-etl","max_age_seconds":300}, {"name":"fx fresh","dag":"trade-etl","node":"enrich_fx","max_age_seconds":120} ]}
