Cloud Pub/Sub & Resilient Connectors (v2.0.0)

Publish to and subscribe from S3, Azure Blob, and Google Cloud Storage; wrap fragile connections in retry/buffer/reconnect logic with one config flag.

Object-store pub/sub

Three new URI schemes work anywhere a kafka:// or rest:// source/destination does:

  • s3://bucket/prefix - Amazon S3 (or MinIO/LocalStack via endpoint_url)
  • azureblob://container/prefix - Azure Blob Storage (requires connection_string)
  • gcs://bucket/prefix - Google Cloud Storage (optional credentials_file)

Each published message becomes one JSON object named prefix/msg-<timestamp_ns>-<uuid>.json; subscribers poll the prefix in order and (by default) delete objects after reading.

Example DAG nodes
"publishers": { "s3_out": { "destination": "s3://my-bucket/dag-output", "region": "us-east-1" } }, "subscribers": { "gcs_in": { "source": "gcs://my-bucket/dag-input", "poll_interval": 2, "delete_on_read": true } }
Config keyDefaultMeaning
poll_interval2Seconds between prefix listings (subscriber)
delete_on_readtrueDelete objects after consumption
region / endpoint_url-S3 only
connection_stringrequiredAzure Blob only
credentials_fileADCGCS only
Resilient connectors

Add "resilient": true to any aerospike://, sql://, rest://, or grpc:// publisher/subscriber to wrap it with:

  • Retry with exponential backoff - retry_max attempts (default 5), starting at retry_backoff_seconds (default 2) and doubling
  • Outage buffering (publishers) - up to buffer_max_messages (default 10,000) messages are held in FIFO order and drained on recovery; new messages never overtake buffered ones
  • Automatic reconnection - the inner connection is rebuilt between retries and after receive errors
Example
"publishers": { "orders_db": { "destination": "sql://orders_table", "resilient": true, "retry_max": 8, "retry_backoff_seconds": 1, "buffer_max_messages": 50000, "db_url": "postgresql://user:pass@db:5432/orders" } }

Every failure is logged with a full stack trace; wrapper state (buffer depth, consecutive errors) is visible in each pub/sub's details() on the DAG details page.
