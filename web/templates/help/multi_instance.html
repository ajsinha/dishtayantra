Multi-Instance Management

Manage several DishtaYantra instances from a single UI (the service-plane / UI-plane split).

What it is

One instance acts as a UI plane: your browser talks only to it, and it proxies to whichever server you select. Other instances are service planes. This is control-plane separation, not clustering — every instance still runs as an independent single-node compute server.

Setup

  1. On the UI-plane instance, set a stable encryption secret (it encrypts trusted-server keys at rest):
    export DY_SECRET_KEY="a-long-stable-passphrase" python3 run_server.py
  2. On each remote instance, mint an API key (Admin → API Keys). Use an admin key for full control or a user key for read-only.
  3. On the UI plane, open Admin → Manage servers (/admin/trusted-servers) and add each remote by id, name, URL, key, and role. The UI plane probes the remote on save and records its version; a bad URL/key is rejected immediately.

Using it

  • Use the navbar server switcher to re-target. The active server is shown as a pill so you always know where you're acting.
  • Re-targets: DAG list/lifecycle, flow time-travel, metrics/health, worker & egress status, and designer deploy.
  • Always local: your login, users, and API keys; plus host-scoped admin (worker pool, language runtimes, maintenance).
  • Fleet view (/fleet): a read-only overview of every server — reachability, version, DAG counts — with one-click switch. An unreachable server shows its own error tile rather than blanking the page.

Security

  • Trusted-server keys are encrypted at rest (Fernet via DY_SECRET_KEY), masked everywhere, and never sent to the browser.
  • Cross-server actions carry X-DY-On-Behalf-Of so the remote's audit log attributes the action to you (advisory only; the remote still authorises on the API key).
  • The UI plane authenticates you before proxying anything — never an open relay. Version compatibility is the admin's responsibility (the UI warns on mismatch but does not enforce).

Notes

  • DY_SECRET_KEY must be stable; rotating it invalidates stored keys (re-enter them). Keep it in the environment, not in config.
  • Prefer HTTPS endpoints; verify_tls can be disabled per entry for self-signed lab setups (not for production).
  • Live views use proxied polling today; a push transport is a planned optimisation.
